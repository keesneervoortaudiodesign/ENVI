<objective>
Build the thin `envi-compute-wasm` cdylib — the browser boundary over the `envi-compute` core — with its two genuinely-new shapes: the OPFS-backed `TensorSink` (`FileSystemSyncAccessHandle`, worker-only) and the threaded-wasm build toolchain (nightly + build-std + atomics, scoped to one npm script). It mirrors the existing `envi-gis-wasm` cdylib discipline: marshalling only, ts-rs boundary DTOs into the single committed `wire.ts`, the exact `=0.2.126` wasm-bindgen pin.

Purpose: This crate is the seam where the browser calls the FORCE-validated engine `solve()` and streams chunks to OPFS. The OPFS sink is a new impl of the existing engine trait (no engine change, D-02). The threaded build must be isolated so it never forces nightly/atomics onto the stable gis-wasm and native builds (Pitfall 1). The rayon pool driver + the worker that drives this crate land in Plan 10-04.

Output: `crates/envi-compute-wasm/` (Cargo.toml + lib.rs boundary + dto.rs + opfs_sink.rs), the `build:wasm:compute` npm script, and the git-ignore/asset/doc wiring.
</objective>
