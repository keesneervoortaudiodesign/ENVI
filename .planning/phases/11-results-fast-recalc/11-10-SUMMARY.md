# Phase 11 Plan 10: Scene-Object Color + Hatch Styling Sweep Summary

**The 9 scene-object kinds now render at full NoizCalc §4.6.3 styling ON TOP of the isophone fill (D-18): a dataviz-validated categorical hue plus a geometry-specific secondary channel — point marker glyphs (disc/ring/diamond) + border, line width/dash, and semi-transparent AREA fills with a SEPARATE runtime-generated hatch (45°/135° arcering) — driven by a display map fully decoupled from Terra Draw's draw-time behaviour, proven by an offline UAT + a Phase-7 draw-journey regression guard.**

## Accomplishments

- **Per-kind display styling + runtime rasters (Task 1).** `objectStyles.ts` encodes the 11-UI-SPEC §Palettes #3 categorical table as a `Record<Kind, KindDisplayStyle>` discriminated union — points carry colour + glyph + size + border, lines carry colour + width (+ dash), areas carry fill colour + fill-opacity + border + a separate hatch id. `hatchPatterns.ts` generates each area hatch (solid-diagonal 45°, dotted/stipple, cross-hatch, sparse 135°) and each point marker glyph as a pure `ImageData`-shaped raster (no DOM canvas → Node-testable, `map.addImage`-consumable). `theme.css` gains `--obj-*` categorical tokens mirroring the objectStyles hex (D-11). 7 unit tests cover 9-kind coverage, the geometry-appropriate secondary channel, within-family colour uniqueness, and non-empty raster pixels.
- **Object layers above the isophone fill (Task 2).** A new `SceneObjectLayers` child in `MapCanvas.tsx` registers the hatch + marker images and adds `fill` (translucent) → `fill` (hatch pattern) → `line` (border, + dashed calc-area frame) → `line` (solid/dashed) → `symbol` (point markers) layers driven by data-driven `match` expressions on `properties.kind`, fed by the canonical `sceneFeatureCollection()`. The isophone (11-06) and difference (11-16) fills now carry `envi-object` in their `SCENE_LAYER_PREFIXES` so they re-insert BELOW the styled objects (D-18). `useTerraDraw.ts` is untouched.
- **Offline UAT + draw-time regression guard (Task 3).** `objectStyling.spec.ts` drives the real bundle offline: it seeds the isophone fill, places all 9 kinds, and asserts (via `objectLayerTelemetry`) that every display layer + hatch/marker image is registered and every object layer sits ABOVE `envi-isophone-fill` in the style order. Its second test replays a Phase-7 draw-time journey (ground_zone containment / partial-cross reject / last-object inheritance) and asserts identical behaviour — the display restyle did not regress draw-time validation.

**2. [Rule 3 - Implementation choice] Hatch/marker generated as pure ImageData rasters, not a DOM `<canvas>`**
- **Found during:** Task 1.
- **Issue:** RESEARCH described a runtime `<canvas>`, but the vitest env is `node` (no DOM/canvas) and the acceptance asserts `hatchPattern` returns an ImageData with size/non-empty pixels in a unit test.

- **Verification:** 7 unit tests green in the node env; the browser UAT confirms the images register + render.
- **Committed in:** `3daccbc` (Task 1 commit).
